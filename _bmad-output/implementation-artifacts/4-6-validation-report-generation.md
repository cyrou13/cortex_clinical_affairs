# Story 4.6: Validation Report Generation

Status: ready-for-dev

## Story

As an RA Manager,
I want to generate Validation Reports and Clinical Benefit Reports in DOCX format,
So that I have submission-ready validation documentation.

## Acceptance Criteria

**Given** a validation study with protocol, imported results, and SOA benchmark mapping
**When** the RA Manager clicks "Generate Report"
**Then** a Validation Report (DOCX) is generated with: protocol, results, SOA comparison, amendment history (FR41)
**And** for MRMC studies, a Clinical Benefit Report is generated (DOCX) (FR42)
**And** reports include protocol version, amendment history (FR37c)
**And** generation runs asynchronously with progress tracking in AsyncTaskPanel
**And** the generated DOCX is available for download
**And** the report follows MDR-compliant formatting (I2)

## Tasks / Subtasks

### Backend / Worker Tasks

- [ ] **T1: Create Validation Report data preparation** (AC: protocol, results, comparison, amendments)
  - [ ] T1.1: Create `apps/workers/src/processors/validation/prepare-report-data.ts` — aggregates all data for the report
  - [ ] T1.2: Gather protocol data: current version, summary, endpoints, sample size justification, statistical strategy
  - [ ] T1.3: Gather amendment history: all ProtocolAmendment records sorted by date (FR37c)
  - [ ] T1.4: Gather results data: active DataImport with computed statistics per endpoint
  - [ ] T1.5: Gather SOA benchmark comparison: ResultsMapping records with MET/NOT_MET status
  - [ ] T1.6: Gather study metadata: study name, type, creation date, linked SOA analysis reference
  - [ ] T1.7: Format data into `ValidationReportData` structure compatible with Carbone template

- [ ] **T2: Implement Validation Report generation** (AC: DOCX generation, FR41)
  - [ ] T2.1: Extend `apps/workers/src/processors/validation/generate-reports.ts` — handle `VALIDATION_REPORT` type
  - [ ] T2.2: Use hybrid engine:
    - Carbone renders base template (`validation-report.docx`) with protocol data, amendments, metadata
    - docx npm generates dynamic sections: results table (variable rows per endpoint), comparison table (color-coded MET/NOT_MET), statistical summary
  - [ ] T2.3: Results table columns: Endpoint, N, Result, 95% CI, Target, Status
  - [ ] T2.4: Comparison table: Endpoint, Result, SOA Benchmark, Status (MET/NOT_MET), Source Reference
  - [ ] T2.5: Amendment history table: Version, Date, Author, Reason, Changes Summary
  - [ ] T2.6: Include document footer: "Generated by CORTEX Clinical Affairs Suite | {date} | Protocol v{version}"

- [ ] **T3: Implement Clinical Benefit Report generation** (AC: MRMC report, FR42)
  - [ ] T3.1: Create or extend template for Clinical Benefit Report (specific to MRMC studies)
  - [ ] T3.2: Clinical Benefit Report includes:
    - Section 1: Study Design (MRMC methodology, reader pool description)
    - Section 2: Reader Analysis (per-reader performance, inter-reader variability)
    - Section 3: Subject Analysis (per-subject results, difficult case analysis)
    - Section 4: Clinical Benefit Assessment (how AI improves reader performance)
    - Section 5: Conclusions and Clinical Significance
  - [ ] T3.3: Generate dynamic MRMC-specific tables using docx npm:
    - Reader performance matrix (reader x metric)
    - Subject difficulty distribution
    - Reader variability statistics (ICC, Kappa)
  - [ ] T3.4: Only available for studies with type MRMC

- [ ] **T4: Create report generation validation** (AC: prerequisites check)
  - [ ] T4.1: Before generation, validate prerequisites:
    - Protocol must be APPROVED or AMENDED (not DRAFT)
    - Active DataImport must exist
    - ResultsMapping must be computed
  - [ ] T4.2: Return clear error if prerequisites not met: `ValidationError` with specific missing prerequisite
  - [ ] T4.3: For Clinical Benefit Report, additionally validate study type is MRMC

- [ ] **T5: Store and track generated reports** (AC: download availability)
  - [ ] T5.1: Upload generated DOCX to MinIO: `validation/{studyId}/reports/{reportType}_{version}_{timestamp}.docx`
  - [ ] T5.2: Create ValidationReport record with: filePath, reportType, status (GENERATING, COMPLETED, FAILED), generatedAt, generatedById
  - [ ] T5.3: Keep history of all generated reports (not just latest) for audit trail
  - [ ] T5.4: Emit domain event `validation.report.generated` with `{ studyId, reportType, reportId, filePath }`

- [ ] **T6: Extend GraphQL layer** (AC: API for report generation and download)
  - [ ] T6.1: Ensure mutation `generateValidationReport(studyId, reportType)` handles VALIDATION_REPORT and CLINICAL_BENEFIT types
  - [ ] T6.2: Add query `validationReports(studyId)` — returns list of all generated reports with download URLs
  - [ ] T6.3: Each report includes a signed download URL (presigned MinIO URL, valid for 1 hour)
  - [ ] T6.4: Apply RBAC: Admin, RA Manager can generate reports; Data Science can view

## Dev Notes

### Technology Stack (Exact Versions)

- **DOCX Generation:** Carbone 5.x + docx 9.x (hybrid engine from Story 4.5)
- **BullMQ:** 5.69.x — queue `validation:generate-report`
- **MinIO:** S3-compatible storage for generated documents
- **Frontend:** React 19.x, Apollo Client 3.x (subscriptions for progress), Tailwind CSS 4.x, shadcn/ui
- **Testing:** Vitest

### Prerequisite Validation

```typescript
interface ReportPrerequisites {
  protocolApproved: boolean; // Protocol status must be APPROVED or AMENDED
  activeImportExists: boolean; // At least one DataImport with ACTIVE status
  resultsMapped: boolean; // ResultsMapping records exist
  isMRMC?: boolean; // For Clinical Benefit Report: study type must be MRMC
}
```

### UX Design Notes

- **Report generation section:** Cards for each report type within the validation study detail page
- **Prerequisites indicator:** Green check or orange warning badge on each card
- **"Generate" button:** Primary variant with download icon — disabled (opacity 50%, tooltip) when prerequisites not met
- **Async progress:** Shown in AsyncTaskPanel (bottom-right) with stages: "Gathering data... 25%", "Rendering template... 50%", "Generating tables... 75%", "Complete!"
- **Toast on completion:** Success toast with [Download] action link
- **Report history:** List of previously generated reports below the generation cards — version, date, file size, download link
- **Download:** Direct browser download via presigned MinIO URL (no intermediate page)
