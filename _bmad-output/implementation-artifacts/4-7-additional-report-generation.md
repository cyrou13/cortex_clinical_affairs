# Story 4.7: Additional Report Generation

Status: done

## Story

As an RA Manager,
I want to generate specialized reports (Algorithmic Fairness, Labeling Validation, FDA 18.CVS, Patch Validation),
So that I have complete validation documentation for all regulatory pathways.

## Acceptance Criteria

**Given** a completed validation study
**When** the RA Manager selects additional report types
**Then** Algorithmic Fairness Report can be generated (DOCX) (FR42a)
**And** Labeling Validation Checklist can be generated (DOCX) (FR42b)
**And** Benefit Quantification Report can be generated (DOCX) (FR42c)
**And** structured output set can be exported (FR42d)
**And** Patch Validation Report with delta analysis can be generated (FR42e)
**And** delta analysis highlights changes between parent study and patch study (FR42f)
**And** FDA 18.CVS (Clinical Validation Study) standalone report can be generated (FR42g)
**And** FDA 18.CVS uses FDA-specific formatting (FR42h)
**And** all reports run asynchronously via the DOCX generation engine

### Delta Analysis Data Structure

```typescript
interface DeltaAnalysis {
  parentStudyId: string;
  patchStudyId: string;
  endpointDeltas: Array<{
    endpointName: string;
    parentResult: number;
    patchResult: number;
    absoluteChange: number;
    relativeChange: number; // percentage
    direction: 'improved' | 'degraded' | 'unchanged';
    statisticallySignificant: boolean;
    pValue?: number;
  }>;
  overallAssessment: 'acceptable' | 'requires_review' | 'not_acceptable';
}
```

### FDA 18.CVS Formatting Differences

- **Page size:** US Letter (8.5" x 11") instead of A4
- **Margins:** 1 inch all sides (FDA standard)
- **Section numbering:** 1., 1.1., 1.1.1. (FDA style)
- **Reference style:** Numbered sequential [1], [2], [3]
- **Cover page:** FDA submission format with 510(k) number field
- **No Annex structure** — flat section layout
- **Footer:** "FDA 18.CVS — Clinical Validation Study | {device_name}"

### UX Design Notes

- **Report cards grouped by category** — use Card component with category headers (text-lg semibold)
- **Conditional visibility** with smooth transitions — reports appear/disappear based on study type
- **"Generate" button:** Primary variant. Disabled (opacity 50%) with tooltip when prerequisites not met
- **Structured Output export:** Different visual treatment — uses a secondary/ghost button with download icon (not Primary)
- **Patch study selector:** Dropdown with StatusBadge per parent study option (locked = green, in-progress = blue)

### Anti-Patterns to Avoid

- Do NOT create separate BullMQ queues per report type — reuse `validation:generate-report` with reportType in job data
- Do NOT duplicate data preparation logic — each report type has its own data preparation module but shares common data fetching
- Do NOT skip progress reporting — every generation job must emit progress events
- No `any` types — use proper discriminated unions for report type data
